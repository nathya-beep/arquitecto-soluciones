## 9. Build Order

{THIS IS THE MOST CRITICAL SECTION. Numbered steps with clear deliverables per step.}

### Build Discipline (the builder MUST follow this)

1. **Read this entire BLUEPRINT.md before writing a single line of code.**
2. **Build in this exact order, block by block.** Do not jump ahead. If a block depends on another, build the dependency first.
3. **Security is NOT a final step.** The security foundation — authentication, session handling, route/endpoint protection, roles & permissions (§8) and rate limiting + sensitive-data rules (§16) — must be implemented as early as its dependencies allow (right after scaffolding + data model + auth service wiring), never bolted on at the end. Every step that adds an endpoint or route MUST also wire in its auth check and (where applicable) its rate limit in the SAME step — never "to be secured later".
4. **After completing each step, stop:** give a short summary of what was built and confirm with the user before starting the next step.
5. **If a security decision is undefined or ambiguous** (an endpoint with no stated auth rule, a field whose sensitivity isn't classified, a missing rate limit on a sensitive action), STOP and ask the user. Never invent it or defer it.

### Security placement rule

The build order below is arranged so the security foundation lands early. Each step lists its **Dependencies** and, when it introduces a route/endpoint/data, its **Security responsibilities** (auth check, role required, rate limit, sensitive-data handling) drawn from §8 and §16.
